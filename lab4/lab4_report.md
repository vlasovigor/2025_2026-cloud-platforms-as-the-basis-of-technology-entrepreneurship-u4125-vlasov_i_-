## Отчет по лабораторной работе "Разработка инфраструктуры MVP AI приложения"  
**Цель работы:** 
Создать прототип AI-приложения с базовой функциональностью.  

## 1. Описание приложения

Приложение должно помогать инвестору быстро получать краткую аналитику по финансовой отчетности компании.

Приложение должно иметь функции:
1. авторизоваться;
2. загрузить финансовый отчет;
3. получить краткую аналитическую сводку;
4. посмотреть историю своих запросов;
5. не запускать повторный анализ, если такой отчет уже был разобран раньше.

Я предполагаю, что в приложении будет использоваться готовая модель через API, а также простой RAG-подход, чтобы модель использовала специальные знания по финансовому анализу.


## 2. Архитектура

Я выбрал **serverless-подход**, потому что по заданию на старте пользователей мало, и нет смысла сразу брать сложные и дорогие сервисы.

Основная логика такая:

1. Пользователь заходит через **Telegram Bot** или **Web UI**.
2. Запрос попадает в **Cloud Run API**.
3. Загруженный отчет сохраняется в **Cloud Storage**.
4. Данные о пользователе, истории и готовых результатах хранятся в **Firestore**.
5. Если нужный анализ уже есть, система берет его из кэша.
6. Если анализа нет, создается задача в **Pub/Sub**.
7. **Cloud Run Worker** обрабатывает отчет.
8. Worker берет материалы из **RAG-базы знаний** и отправляет контекст в **Vertex AI / Gemini**.
9. Готовый результат сохраняется в Firestore.
10. Пользователь получает готовую аналитику.

## 3. Обоснование использования сервисов

### Cloud Run
Я выбрал Cloud Run, потому что это удобно для небольшого приложения:

- не нужно поднимать сервер вручную;
- сервис может масштабироваться автоматически;
- при маленькой нагрузке это дешево;
- есть free tier.

### Cloud Storage
Cloud Storage подходит для хранения отчетов, потому что отчеты — это файлы, и хранить их в базе данных неудобно.

### Firestore
Firestore я выбрал для хранения:

- пользователей;
- истории запросов;
- результатов анализа;
- кэша.

Эта база подходит для MVP, потому что она простая и не требует администрирования.

### Pub/Sub
Pub/Sub нужен, чтобы анализ отчета не выполнялся прямо в пользовательском запросе. Это полезно, если обработка занимает заметное время.

### Vertex AI / Gemini
Я выбрал готовую модель, потому что по условию не нужно обучать свою модель. Это проще и реалистичнее для прототипа.

### RAG
RAG нужен, чтобы AI использовал дополнительные знания по финансовому анализу, а не отвечал только на основе общего знания модели.


## 4. Три состояния приложения

## 4.1. Начальный этап

**Нагрузка:** 10 пользователей в месяц.

На этом этапе я бы использовал:

- Telegram Bot;
- Cloud Run API;
- Cloud Storage;
- Firestore;
- Vertex AI / Gemini.

### Почему так
На старте важно сделать дешево и просто. Сложные сервисы вроде Kubernetes тут не нужны.

---

## 4.2. Тестирование партнерами

**Нагрузка:** 100–200 пользователей в месяц.

На этом этапе я бы добавил:

- Pub/Sub;
- отдельный Cloud Run Worker;
- кэширование результатов;
- базовый мониторинг.

### Почему так
Когда пользователей становится больше, лучше разделить API и обработку отчета. Тогда пользователь не будет долго ждать ответа в одном запросе.

---

## 4.3. Production

**Нагрузка:** около 30 000 пользователей в месяц.

На этом этапе архитектура остается такой же по идее, но становится более полной:

- Web UI + Telegram Bot;
- Cloud Run API;
- Cloud Run Worker;
- Cloud Storage;
- Firestore;
- Pub/Sub;
- RAG-база знаний;
- Secret Manager;
- Cloud Logging.

### Почему так
Мне кажется, это хороший вариант, потому что архитектура уже знакомая, но при этом она масштабируется. Не нужно полностью переделывать приложение при росте нагрузки.
